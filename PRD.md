## 1. 产品概述

### 1.1 产品定位

LearnFlow 是一款面向个人学习者的 **AI 驱动学习执行系统**。它不是"学什么"的推荐工具，而是"怎么学、怎么坚持"的执行引擎。

### 1.2 解决的核心问题

| 问题 | LearnFlow 的解法 |
|---|---|
| 制定了学习计划却坚持不了 | 每日推送 + 强制解锁机制，不完成就不能前进 |
| 看了很多教程但没真正理解 | 强制输出闭环：每步必须提交输出并通过 AI 评审 |
| 不知道学习重点在哪 | 二八法则标注核心20%，聚焦高价值内容 |
| 学完就忘，没有体系化 | 学习阶梯5级递进 + 速查表压缩 + 费曼技巧输出 |
| 学习资源质量参差不齐 | 噪音筛选机制，每周精选3-5个资源并标注推荐理由 |

### 1.3 目标用户

个人学习者，尤其是自学编程、AI、新技术的开发者。特征：有一定自学能力但缺乏系统性，容易半途而废或陷入"假性学习"（只看不练）。

### 1.4 产品差异化

与 Coursera、Udemy 等课程平台的本质区别：LearnFlow 不提供课程，而是**生成个性化的学习执行计划并强制用户输出**。核心理念是"教是最好的学"——每一步都要求用户用自己的话总结，通过 AI 扮演"严格导师"角色进行评审。

## 2. 核心功能

### 2.1 学习计划创建（PlanWizard）

**用户故事：** 作为学习者，我想通过简单的向导创建一个完整的学习计划，而不是自己规划每周每天要学什么。

**功能描述：**

4步向导式创建流程：

1. **选择主题与目标** — 输入学习主题（如"机器学习"），选择学习目标（入门了解 / 系统掌握 / 项目实战 / 深度精通 / 探索前沿），可添加重点方向标签
2. **评估当前水平** — 三选一卡片：入门（零基础）、进阶（有基础想深化）、高级（已有经验想突破）
3. **安排学习时间** — 设置每周学习小时数（3-30h 滑块）、选择学习日（周一到周日多选）、设定开始日期
4. **确认创建** — 预览计划摘要，确认后系统调用 LLM 生成完整计划

**计划生成逻辑（plan_engine.py）：**

- 根据用户水平和目标估算总周数（入门16周、进阶12周、高级8周，可按时间调整）
- 构建5级学习阶梯，将周数按比例分配：认知启蒙(15%) → 基础构建(25%) → 实践应用(30%) → 深化拓展(20%) → 体系输出(10%)
- 调用 LLM 分批生成每周内容（每批4周，避免 JSON 截断）
- 为每周生成：学习摘要、核心20%内容、3-5个精选资源（含至少1个文档资源）、实践任务、测试题
- 当 LLM 不可用时，回退到 topic_library.py 预设内容

**数据产出：** 一个 LearningPlan + N 个 LearningStep + M 个 Milestone

### 2.2 每日学习步骤（LearningStep）

**用户故事：** 作为学习者，我想每天打开 APP 就看到今天该做什么，有清晰的任务指引和配套资源。

**步骤类型：**

| 类型 | 说明 | 时长 | 出现频率 |
|---|---|---|---|
| study（系统学习） | 核心概念学习 + 配套资源 | 60min | 每周2-3次 |
| project（动手实践） | 基于本周所学完成实践任务 | 90min | 每周1次 |
| deep_project（深度实战） | 综合性项目任务 | 120min | 每周0-1次（5天以上） |
| output（总结输出） | 用自己的话总结本周所学 | 60min | 每周1次 |
| cheat_sheet（速查表压缩） | 将阶段所学压缩为速查表 | 45min | 等级过渡周 |
| test（阶段测试） | 通过后方可解锁下一等级 | 30min | 按需 |

**步骤内容结构：**

每个步骤包含：标题、学习内容（200-300字中文摘要）、核心20%标注（高亮显示）、精选资源列表（视频/文档/文章，含平台、难度、耗时、推荐理由）、实践任务描述。

**资源质量要求：**

- 每周至少1个 doc 类型文档资源（官方文档、技术规范等）
- 视频资源优先推荐B站（bilibili.com）
- 文档资源优先官方文档
- URL 必须真实可访问
- 每个资源标注：type、title、url、platform、level（入门/进阶/高级）、duration、why

### 2.3 强制输出闭环（核心机制）

**用户故事：** 作为学习者，我想被"逼着"真正理解知识，而不是假装学会了就跳过。

**流程：**

```
步骤状态：locked → available → in_progress → [提交输出] → review_passed → completed
                                                          ↓
                                                    review_failed → [修订重提] → review_passed
```

1. 用户点击"开始学习"，步骤状态从 available 变为 in_progress
2. 学习完成后，用户点击"提交输出"，在文本框中用自己的话总结所学
3. 系统调用 AI 评审服务（reviewer.py），评分 0-100
4. ≥70分：通过，自动解锁下一步，生成成就/里程碑检查
5. <70分：未通过，显示反馈和改进建议，用户需修订后重新提交

**AI 评审三方法：**

1. **苏格拉底追问** — 每次评审附带1个深挖问题，检验用户是否真正理解（不是记住）
2. **费曼技巧检测** — 占总分25%权重，检测用户是否只是堆砌术语而没有真正理解。如果检测到术语堆砌，要求用户用"给10岁小孩解释"的方式重新描述
3. **速查表生成** — 通过评审时自动生成该步骤的知识速查表，便于长期记忆和复习

### 2.4 学习阶梯（5级递进）

**用户故事：** 作为学习者，我想有一个清晰的进阶路径，知道自己在什么阶段、下一步要达到什么水平。

**5级结构：**

| 等级 | 名称 | 占比 | 核心关注 | 过关标准 |
|---|---|---|---|---|
| 1 | 认知启蒙 | 15% | 基本概念和术语 | 能用自己的话解释该领域是什么 |
| 2 | 基础构建 | 25% | 核心原理和基础技能 | 能独立完成入门级任务 |
| 3 | 实践应用 | 30% | 项目实战和最佳实践 | 能独立完成中等复杂度项目 |
| 4 | 深化拓展 | 20% | 高级特性和底层原理 | 能解释设计决策的 trade-off |
| 5 | 体系输出 | 10% | 知识体系化和教学 | 能写教程或指导初学者 |

每个等级包含：核心知识点、常见错误、过关标准。等级过渡周自动插入速查表压缩和阶段测试环节。

### 2.5 仪表盘（Dashboard）

**用户故事：** 作为学习者，我想一眼看到今天该做什么、整体进度如何、最近有什么成就。

**展示内容：**

- 活跃计划卡片：进度条、统计数据（连续天数、已完成步骤数、当前周数）
- 今日任务卡片：步骤类型标签、耗时、核心20%高亮、"开始学习"按钮
- 即将进行的步骤列表
- 里程碑列表（带周数标记）
- 成就徽章展示（按稀有度分级：普通/稀有/史诗/传说）

### 2.6 个人档案（Profile）

**用户故事：** 作为学习者，我想回顾自己的学习历程，看到量化的成长和进步。

**展示内容：**

- 4宫格统计：总计划数、已完成步骤、总学习时长、总成就数
- 连续学习天数卡片
- 计划进度列表（带进度条，可点击进入学习）
- 最近完成活动（含 AI 评审分数）

### 2.7 通知提醒

**用户故事：** 作为学习者，我想在每天学习时间收到提醒，不忘记学习。

**支持渠道：**

| 渠道 | 实现 | 触发方式 |
|---|---|---|
| Windows Toast | win11toast 原生通知 | 每日 8:00 定时任务 |
| 飞书（Lark） | lark-cli | 可配置提醒时间 |
| 钉钉 | dws CLI | 可配置提醒时间 |

每个渠道可独立配置启用/禁用、提醒时间，支持测试发送。

### 2.8 LLM 配置管理

**用户故事：** 作为用户，我想在 APP 内直接配置 LLM API，不需要修改文件。

**功能：** PC 端管理后台提供 LLM 配置表单（API Base URL、API Key、Model Name），保存后运行时立即生效并持久化到 .env 文件。API Key 返回时脱敏（仅显示 has_key: true/false），更新时空值保留原 Key。

### 2.9 Skill 设计理念 — 将学习方法论变成可执行代码

LearnFlow 的核心竞争力不在于 UI 或功能数量，而在于将 6 个经过验证的学习方法论深度嵌入后端引擎，从"写在文档里的方法"变成"自动执行的代码"。

**设计哲学：** 采用"执行引擎型"Skill 设计 — 用户输入目标，系统自动输出完整方案，而非让用户手动选择工具和方法。

#### Skill 1：学习计划引擎（learning-plan-engine）

将模糊的"我想学机器学习"转化为每日可执行的学习计划。嵌入 3 个学习方法：

- **学习阶梯（Learning Ladder）** — 代码中 `LADDER_LEVELS` 定义了 beginner/intermediate/advanced 三套 5 级递进结构。每级包含核心知识点、常见错误、过关标准。系统自动将总周数按比例分配到 5 个等级（15%/25%/30%/20%/10%），等级过渡周自动插入速查表压缩和阶段测试。
- **二八法则（Pareto Principle）** — `CORE_20_TEMPLATES` 为 5 种学习目标定义了核心聚焦方向。每周生成内容时标注"核心20%"，让用户明确知道哪些知识值得深钻、哪些只需了解。
- **噪音中找信号（Signal in Noise）** — LLM prompt 强制要求每周精选 3-5 个资源，必须包含文档类型资源，视频优先B站，每个资源标注难度、耗时和推荐理由。从海量学习资源中筛选出真正高信噪比的内容。

#### Skill 2：学习输出评审引擎（learning-output-reviewer）

扮演严格导师角色，确保用户真正理解而非假装学会。嵌入 3 个学习方法：

- **苏格拉底追问（Socratic Questioning）** — 每次评审附带 1 个深挖问题，追问用户"为什么"和"如果不是这样会怎样"，检验理解深度而非记忆程度。
- **费曼技巧（Feynman Technique）** — 占总评分权重 25%。检测用户是否用简单语言解释（`simple_language`）、是否使用了类比（`has_analogy`）、是否堆砌术语（`jargon_overuse`）。如果检测到术语堆砌，要求用"给 10 岁小孩讲"的方式重新描述。
- **速查表压缩（Cheat Sheet Compression）** — 通过评审时自动生成结构化速查表建议，将核心知识压缩为一页纸，便于长期记忆和快速复习。

评审采用 5 维度加权评分：理解深度(30%) + 准确性(25%) + 完整性(20%) + 原创表达(15%) + 实践关联(10%)。≥70 分解锁下一步，<70 分返回 3 条改进建议。LLM 不可用时自动回退到规则版评审。

#### 四层 Skill 协作体系

```
方法论源头     ai-learning-coach（6个学习方法 + 18种场景变体）
                  ↓ 拆分                 ↓ 拆分
核心 Skill    learning-plan-engine    learning-output-reviewer
              (阶梯+二八+噪音)         (苏格拉底+费曼+速查)
                  ↓ API                    ↓ API
执行编排层     ai-learning-engine（串联计划生成 + 评审闭环）
                  ↓
用户交互       LearnFlow APP（Dashboard → Wizard → 学习 → 输出 → 评审）
```

4 个 Skill 分工明确：`ai-learning-coach` 提供方法论理论框架，两个核心 Skill 分别嵌入后端引擎处理"计划生成"和"输出评审"，`ai-learning-engine` 作为编排层串联完整学习闭环。

## 3. 非功能需求

### 3.1 性能

- 计划创建（含 LLM 生成）：16周计划约 2-3 分钟（分批生成，每批4周）
- 步骤查询：< 200ms
- AI 评审：< 30 秒
- 仪表盘加载：< 500ms

### 3.2 兼容性

- **PC端：** Chrome、Edge、Firefox 最新版，宽度 ≥ 768px
- **移动端：** iOS Safari、Android Chrome，宽度 < 768px
- 设备检测：768px 断点，移动端和 PC 端使用不同的路由和布局

### 3.3 数据安全

- 所有数据存储在本地 SQLite，不上传到云端
- LLM API Key 存储在 .env 文件，API 返回时脱敏
- .env 和数据库文件均被 gitignore

### 3.4 国际化

- 支持中文（zh-CN）和英文（en-US）双语
- 所有 UI 文案通过 vue-i18n 管理
- LLM 生成的内容默认中文

## 4. 技术约束

### 4.1 已知限制

- **SQLite 并发：** 单文件数据库，不支持高并发写入。适合个人使用，不适合多用户
- **LLM 依赖：** 计划生成和 AI 评审依赖外部 LLM API。不可用时回退到预设内容（目前覆盖 React、Python、LLM 三个主题）
- **无用户认证：** 单用户设计，无登录/注册机制
- **手动迁移：** 数据库 Schema 变更通过 ALTER TABLE 手动迁移，无 ORM 自动迁移工具
- **Windows 专属通知：** Windows Toast 通知仅支持 Windows 系统

### 4.2 部署约束

- 后端默认端口 8001（8000 可能被其他服务占用）
- uvicorn --reload 在 Windows 上可能残留旧进程，部署前需手动清理
- MiMo V2.5 模型名区分大小写，必须使用全小写 `mimo-v2.5`

## 6. 关键指标

| 指标 | 定义 | 目标 |
|---|---|---|
| 日活完成率 | 每日完成至少1个步骤的天数 / 总天数 | ≥ 80% |
| 输出通过率 | AI 评审一次通过（≥70分）的比例 | 60-80% |
| 计划完成率 | 完整完成一个学习计划的比例 | ≥ 50% |
| 平均评审分 | 所有输出的 AI 评审平均分 | 75-85 |
| 连续学习天数 | 最长连续每天学习的天数 | 持续增长 |
